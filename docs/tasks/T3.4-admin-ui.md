# T3.4 — System Admin console (UI)

**Branch:** `feat/T3.4-admin-ui`  ·  **Depends on:** T3.1, T2.3, T2.5  ·  **Status:** ⬜

## Context
The System Admin's screens: view all (submitted) requests, inspect any request's workflow history,
and manage the sponsorship-type lookup.

## Scope
**In:**
- All-requests view (submitted only — drafts excluded per B5), with filters + pagination.
- Request detail + full workflow history (read-only; admin is not in the approval chain — B3).
- Sponsorship-type management UI (list/create/edit/delete) backed by T2.3.

**Out:** approval actions (admin can't approve — B3); requestor/reviewer screens (T3.2/T3.3).

## References
- HLD §9, §7. Clarifications B3, B5, C2, D1. Backend: T2.3 (types), T2.5 (all-requests/history).
- `docs/best-practices/react-best-practice-rules.md` (§16 four states, §1 focused components).

## Roadmap (high-level)
1. Build all-requests view (filters + pagination) from T2.5; ensure no drafts shown (B5).
2. Build request detail + history (read-only).
3. Build sponsorship-type CRUD UI (forms + table) against T2.3; handle delete-in-use feedback.

## Acceptance Criteria (Definition of Done — also workflow.md §6)
- [ ] Typecheck/ESLint/Prettier/build clean.
- [ ] Admin sees all submitted requests (no drafts), can view any history, cannot approve/reject (B3).
- [ ] Sponsorship-type CRUD works with clear validation/error feedback.

## Test Expectations
- Component tests (RTL): all-requests filtering/pagination; type CRUD happy + validation; no approve controls present for admin.

## Best-Practices Checklist (subset)
- [ ] Focused components; four UI states (react §1,16).
- [ ] No draft leakage (B5); admin scope respected (B3).
