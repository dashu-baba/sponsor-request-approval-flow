# T2.5 — Workflow history + role-scoped queues/lists

**Branch:** `feat/T2.5-history-and-queues`  ·  **Depends on:** T2.2  ·  **Status:** ✅

## Context
Read-side endpoints that power the reviewer queues and audit view: the per-request history and the
role-scoped lists (manager queue, finance queue, admin all-requests), with correct visibility rules.

## Scope
**In:**
- `GET /requests/{id}/history` — visible to owner / relevant reviewer / SystemAdmin.
- Role-scoped list queries: Manager sees `PendingManagerApproval`; FinanceAdmin sees
  `PendingFinanceReview`; SystemAdmin sees **all submitted** requests (drafts excluded — B5);
  Requestor sees own (already in T2.1, extend if needed). Shared queues, no per-user routing (B1),
  not department-scoped (B2).
- Pagination + filtering; AsNoTracking + DTO projection.

**Out:** transitions (T2.2), frontend (T3.3/T3.4), type CRUD (T2.3).

## References
- HLD §7 (RBAC scoping), §8 (API), §15 (B1, B2, B5, D1). Clarifications B1, B2, B5, D1.
- `docs/best-practices/postgres-efcore-best-practice-rules.md` (§18,19,22 projection/no-tracking/pagination).

## Roadmap (high-level)
1. Implement history query with visibility checks (owner/reviewer/admin).
2. Implement role-scoped list queries with the correct status filters + drafts-excluded for admin (B5).
3. Add pagination/filtering; project to DTOs; AsNoTracking.
4. Thin endpoints behind role policies.

## Acceptance Criteria (Definition of Done — also workflow.md §6)
- [ ] Build/format/test clean.
- [ ] Manager/Finance queues return exactly their stage; admin sees all submitted (no drafts, B5).
- [ ] History endpoint enforces visibility; lists are paginated + projected.

## Test Expectations

  admin sees submitted-not-drafts; history visibility honored (owner/reviewer/admin allowed, others 403).
