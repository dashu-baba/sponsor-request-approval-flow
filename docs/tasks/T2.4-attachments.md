# T2.4 — Attachments → MinIO

**Branch:** `feat/T2.4-attachments`  ·  **Depends on:** T2.1  ·  **Status:** ✅

## Context
Optional supporting-document upload for a request, streamed to MinIO (S3-compatible) via the AWS
SDK. Must be implemented without GC pressure — stream, never buffer into memory.

## Scope
**In:**
- Upload endpoint `POST /requests/{id}/attachments` (owner only) + list/download metadata.
- Multiple attachments; validate content-type/extension + per-file size cap (e.g. 10 MB); generate
  server-side object keys (never trust client filenames).
- Stream straight to MinIO via the AWS SDK (`PutObjectRequest`/`TransferUtility`); **no `byte[]`/
  `MemoryStream`**; enforce size limits at app + (later) proxy.
- `Attachment` rows persisted (object key, filename, content-type, size).

**Out:** workflow (T2.2), frontend upload UI (T3.2), TLS/proxy limits config (T4.3 sets nginx `client_max_body_size`).

## Roadmap (high-level)
1. Add MinIO/S3 client config (endpoint/keys from env) + a storage abstraction (port in Application, adapter in Infrastructure).
2. Implement upload handler: validate type/size, generate object key, **stream** `OpenReadStream()` → storage, persist `Attachment`.
3. Set `MultipartBodyLengthLimit`/Kestrel limit to the cap; reject oversize early.
4. Endpoints: upload (owner), list attachments, download/metadata; problem-details errors.

## Acceptance Criteria (Definition of Done — also workflow.md §6)
- [x] Build/format/test clean.

- [x] Type/size validated; oversize rejected; object keys server-generated; owner-only.
- [x] Attachment metadata persisted and listable.
