# T3.3 — Manager + Finance approval queues (UI)

**Branch:** `feat/T3.3-approvals-ui`  ·  **Depends on:** T3.1, T2.2, T2.5  ·  **Status:** ⬜

## Context
The reviewer experience: managers and finance admins see their pending queue, open a request, and
approve/reject with remarks.

## Scope
**In:**
- Manager queue (PendingManagerApproval) and Finance queue (PendingFinanceReview), role-aware.
- Request detail with approve / reject actions; **remarks required on reject** (A5), optional on approve.
- Optimistic-concurrency feedback: if the backend returns 409 (someone else acted), show a clear
  "already actioned / refresh" message.
- Status/history visibility per role.

**Out:** requestor screens (T3.2), admin console (T3.4), the transition backend (T2.2).

## References
- HLD §9, §6. Clarifications A5, B1, B2, B4. Backend: T2.2 (transitions), T2.5 (queues/history).
- `docs/best-practices/react-best-practice-rules.md` (§16 four states, §10 server state, §19 accessibility).

## Roadmap (high-level)
1. Build the manager and finance queue views (role-scoped lists from T2.5).
2. Build the review detail with approve/reject; enforce remarks-on-reject in the form (mirror A5).
3. Handle 409 concurrency + 403 gracefully; invalidate queries on success.
4. Show history timeline for context.

## Acceptance Criteria (Definition of Done — also workflow.md §6)
- [ ] Typecheck/ESLint/Prettier/build clean.
- [ ] Manager/Finance see only their stage; approve advances, reject (with required remarks) terminates.
- [ ] 409 concurrency and 403 are surfaced clearly; lists refresh after actions.

## Test Expectations
- Component tests (RTL): queue rendering; approve flow; reject requires remarks; 409/403 handling.

## Best-Practices Checklist (subset)
- [ ] Four UI states; accessible actions/dialogs (react §16,19).
- [ ] Server state invalidation on mutation; typed API hooks (react §10,15).
