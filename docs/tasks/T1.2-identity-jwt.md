# T1.2 — Identity + JWT authentication

**Branch:** `feat/T1.2-identity-jwt`  ·  **Depends on:** T1.1  ·  **Status:** 🔄 (resolve — review feedback addressed, awaiting re-review)

## Context
Add authentication and the role model so RBAC can be enforced. JWT access tokens + an httpOnly
refresh cookie back the React SPA; role claims drive authorization policies used by every later feature.

## Scope
**In:**
- ASP.NET Core Identity (Postgres-backed) with the four roles: `Requestor`, `Manager`,
  `FinanceAdmin`, `SystemAdmin`; each user holds exactly one role (B6).
- Auth endpoints: `POST /auth/login`, `POST /auth/refresh`, `POST /auth/logout`, `GET /me`.
- Short-lived JWT access token + refresh token in `HttpOnly`/`Secure`/`SameSite` cookie; token
  validation (issuer/audience/lifetime/signature); role claims.
- Authorization policies per role; `JwtOptions` bound + validated on start; secrets from env.

**Out:** request features (T2.x), seeding accounts (T1.3 seeds users into this model), frontend (T3.1),
public self-registration (excluded — B6).

## References
- HLD §2 (auth), §7 (RBAC). Clarifications: B6 (no self-registration, one role/user).
- `docs/best-practices/security-best-practice-rules.md` (§3 hashing, §4 token storage, §5 token validation, §6 least privilege).
- `docs/best-practices/aspnetcore-best-practice-rules.md` (§11–13 authz, strongly-typed options).

## Roadmap (high-level)
1. Add Identity + JWT bearer packages (CPM); extend `AppDbContext` for Identity (or a dedicated context) + migration.
2. Configure Identity (password hashing, role store) and the four roles.
3. Implement login (issue access token + set refresh cookie), refresh (rotate), logout (clear cookie), `/me`.
4. Add JWT validation + authorization policies; bind & validate `JwtOptions`.
5. Add `.env.example` keys for JWT secret + lifetimes; never commit real secrets.

## Acceptance Criteria (Definition of Done — also workflow.md §6)
- [ ] Build/format/test clean.
- [ ] Login returns a valid JWT + sets an httpOnly refresh cookie; refresh rotates; logout clears it.
- [ ] Endpoints enforce role policies; unauthorized → 401, forbidden → 403.
- [ ] Passwords hashed by Identity; no secrets in the repo.

## Test Expectations
- Unit: token generation/validation, policy/role mapping.
- Integration (Testcontainers): login → access protected endpoint → refresh → logout happy path,
  plus 401/403 negative cases.

## Best-Practices Checklist (subset)
- [ ] Server-side authN/authZ; least privilege (security §1–6).
- [ ] Refresh token in httpOnly cookie, not localStorage (security §4).
- [ ] Strongly-typed validated options; no secrets in code (aspnetcore §13–14).
