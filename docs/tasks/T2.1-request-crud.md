# T2.1 — Sponsorship request CRUD + draft

**Branch:** `feat/T2.1-request-crud`  ·  **Depends on:** T1.3  ·  **Status:** ✅

## Context
The requestor's core experience (backend): create/save-draft/edit-while-draft/view-own/list, with
CQRS handlers, validation, mapping, and ownership RBAC. This is the first feature slice and the base
for the workflow engine (T2.2).

## Scope
**In:**
- Commands/queries (MediatR): create request, update draft, get by id, list own (paged).
- Request/response DTOs as records (csharp §14); AutoMapper profiles; FluentValidation validators.
- Rules: editable **only while Draft** (A1); `RequestorName` auto-filled from the logged-in user,
  read-only (C1); amount `> 0` (C3); event date today-or-later (C4); sponsorship type required (C2).
- Ownership enforcement: a requestor acts only on their own requests.
- Endpoints: `GET/POST/PUT /requests`, `GET /requests/{id}` (thin; validation at boundary).

**Out:** workflow transitions (submit/cancel/approve/reject → T2.2), attachments (T2.4), admin/type
CRUD (T2.3), reviewer queues (T2.5), frontend (T3.x).

## Roadmap (high-level)
1. Define DTOs (records) + validators for create/update.
2. Implement create + update-draft commands (enforce A1 edit-only-in-Draft; set RequestorName/Id server-side per C1).
3. Implement get-by-id + list-own queries (AsNoTracking, project to DTO, paginate).
4. Add thin endpoints with role policy (Requestor) + ownership checks; problem-details errors.
5. Wire AutoMapper profiles; validation pipeline behavior.

## Acceptance Criteria (Definition of Done — also workflow.md §6)
- [ ] Build/format/test clean.
- [ ] Create/save-draft/edit-draft/view-own/list work; editing a non-Draft request is rejected (A1).
- [ ] RequestorName/Id come from the token, not the client (C1); amount/date/type validated (C2–C4).
- [ ] List is paginated and projects to DTOs (no full-entity over-fetch).

## Test Expectations
- Unit: validators (amount, date, required type), edit-only-in-Draft rule.
- Integration: create → list-own → get → edit-draft happy path; edit-after-submit rejected; cross-user access forbidden.

## Best-Practices Checklist (subset)
- [ ] Thin endpoints; boundary validation; record DTOs; no entity exposure (aspnetcore §1,4,5).
- [ ] AsNoTracking + projection + pagination (postgres §18,19,22).
- [ ] No mass-assignment of server-owned fields (security §10).
