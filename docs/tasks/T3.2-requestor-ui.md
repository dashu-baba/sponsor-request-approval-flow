# T3.2 — Requestor experience (UI)

**Branch:** `feat/T3.2-requestor-ui`  ·  **Depends on:** T3.1, T2.1, T2.4  ·  **Status:** ⬜

## Context
The requestor's screens: create/edit a request (with draft), view my requests, see a request's
detail + history, and upload supporting documents.

## Scope
**In:**
- Create/edit form (React Hook Form + Zod mirroring backend rules: required fields, amount > 0,
  event date today-or-later); RequestorName shown read-only (from `/me`).
- Save-as-draft vs submit actions; edit allowed only while Draft (mirror A1; rely on backend as source of truth).
- My-requests list (paginated, status badges); request detail view with workflow history timeline.
- Supporting-document upload (optional, multiple, client-side type/size hints; backend enforces).

**Out:** approval queues (T3.3), admin (T3.4); the transition actions for reviewers (T3.3).

## References
- HLD §9. Clarifications A1, C1, C3, C4, C5. Backend: T2.1 (CRUD), T2.4 (attachments), T2.5 (history).
- `docs/best-practices/react-best-practice-rules.md` (§14 forms, §16 four states, §10 server state, §11 Zod).

## Roadmap (high-level)
1. Build the request form (RHF + Zod) with create + edit-draft modes; map server validation errors.
2. Implement my-requests list (TanStack Query, pagination, status badges).
3. Implement request detail + history timeline.
4. Add attachment upload UI (multiple, progress, type/size hints) calling T2.4.
5. Wire submit/cancel actions available to the requestor.

## Acceptance Criteria (Definition of Done — also workflow.md §6)
- [ ] Typecheck/ESLint/Prettier/build clean.
- [ ] Requestor can create, save draft, edit a draft, submit, cancel, and view own requests + history.
- [ ] Form validation mirrors backend; non-Draft requests aren't editable in the UI.
- [ ] Attachment upload works; loading/empty/error/success handled.

## Test Expectations
- Component tests (RTL): form validation paths; list rendering states; submit/cancel actions; upload happy + error.

## Best-Practices Checklist (subset)
- [ ] RHF + Zod forms; four UI states; stable list keys (react §14,16,8).
- [ ] Server state via TanStack Query; API access behind typed hooks (react §10,15).
