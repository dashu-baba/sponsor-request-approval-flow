# T5.3 — Admin audit (SystemAdmin operations trail)

**Branch:** `feat/T5.3-admin-audit`  ·  **Depends on:** T2.1, T2.3, T2.4, T5.1, T5.2  ·  **Status:** ✅

## Context

**Workflow history** (`WorkflowHistory`, `GET /requests/{id}/history`, History tab) is a **business**
requirement (D1): immutable status-transition trail for requestors and reviewers. It is **done** and
**must not change**.

**Admin audit** is a separate concern: a SystemAdmin-only operations/security trail for mutating
actions that today leave no durable record (request create/update, attachment upload, sponsorship-type
CRUD, user creation, auth events). Security best practice §20 requires *“who did what and when”* for
sensitive operations — that is what this task adds.

**Strict isolation:** workflow history and admin audit are **unrelated**. Separate tables, write
paths, APIs, and UI. No FK, join, migration, merged timeline, or duplicate writes on workflow
transitions (submit/cancel/approve/reject stay in `WorkflowHistory` only).

## Scope
**In:**

### Backend — domain & persistence
- New **`AuditEvent`** entity + EF configuration + migration (`audit_events` table only — do **not**
  touch `workflow_history`):
  - `id`, `occurred_at` (timestamptz), `actor_id`, `action`, `category`, `resource_type`,
    `resource_id`, `summary`, `metadata` (jsonb, nullable).
  - Append-only / immutable (no UPDATE/DELETE on audit rows).
  - Indexes: `(occurred_at DESC)`, `(actor_id, occurred_at DESC)`, `(action, occurred_at DESC)`,
    `(resource_type, resource_id, occurred_at DESC)`; optional jsonb index on
    `metadata->>'requestId'` for admin filtering.
- Application port **`IAuditService.RecordAsync(AuditRecord)`**; infrastructure writes to
  `audit_events` in the **same `SaveChanges` transaction** as the business mutation.

### Backend — audit writes (handlers / services)
Wire `IAuditService` on successful mutations only. **Do not** call from
`TransitionRequestCommandHandler`.

| Action code | Category | Trigger |
|-------------|----------|---------|
| `request.created` | Request | `CreateRequestCommand` |
| `request.updated` | Request | `UpdateDraftRequestCommand` |
| `attachment.uploaded` | Attachment | `UploadAttachmentCommand` |
| `sponsorship_type.created` | SponsorshipType | Create command |
| `sponsorship_type.updated` | SponsorshipType | Update command |
| `sponsorship_type.deactivated` | SponsorshipType | Soft-delete command |
| `user.created` | User | `AuthService.CreateUserAsync` |
| `auth.login` | Auth | Successful login |
| `auth.logout` | Auth | Logout |
| `auth.profile_updated` | Auth | Profile update |
| `auth.password_changed` | Auth | Password change |

**Metadata rules:** action-specific jsonb; **never** passwords, tokens, or file bytes. Draft updates:
**changed field names only** (not values). Attachment upload: include loose `requestId` in metadata
(not an FK). User create: email, role, displayName.

**Serilog only (not DB audit):** failed login, token refresh, read-only GETs, attachment downloads.

### Backend — admin read API
- `GET /audit` — paginated global feed; `RequireAuthorization(AuthorizationPolicies.SystemAdmin)`.
- Query filters: `page`, `pageSize`, `action`, `category`, `actorId`, `from`, `to`, `resourceType`,
  `resourceId`, optional `requestId` (metadata filter).
- Returns `PagedResult<AuditEventDto>` (actor display name via join/lookup).
- **`GET /requests/{id}/history` unchanged** — no new request-scoped audit route for non-admins.

### Frontend — Admin audit page
- Route `/admin/audit` (SystemAdmin nav entry beside Users / Sponsorship Types).
- Paginated table: occurred time, actor, action, category, resource, summary.
- Filters aligned with API (action, category, actor, date range, optional requestId).
- Loading / empty / error via existing admin patterns (`QueryStatePanel`, etc.).
- **No audit UI** for Requestor / Manager / FinanceAdmin. Request detail History tab **unchanged**.

## Design decisions (call out for review)
- **Isolation from workflow history** — two stores, zero relationship. Admin investigating a request
  uses History tab (workflow) and Audit page (filter `requestId` in metadata) as **separate screens**.
- **Workflow transitions excluded from audit** — avoids duplicate records and keeps business vs admin
  concerns clean.
- **SystemAdmin-only read path** — audit is an admin capability, not a requestor/reviewer feature.
- **Field names only on draft updates** — privacy + YAGNI; no value diffing.
- **Same-transaction writes** — audit row rolls back if business mutation fails.

## Test Expectations
- **Unit:** `IAuditService` / metadata serialization; optional validator for list query params.

  `audit_events` count unchanged for that action.

  navigate to route.
