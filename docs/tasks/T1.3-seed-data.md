# T1.3 — Seed data

**Branch:** `feat/T1.3-seed-data`  ·  **Depends on:** T1.2  ·  **Status:** ⬜

## Context
Seed the database so reviewers can log in and exercise approvals immediately: one account per role,
sponsorship types, and sample requests spanning every workflow status.

## Scope
**In:**
- Idempotent seeder run once after migration (via the `migrator`/startup seeding step).
- Four documented test accounts (one per role) with known credentials (documented in README, T4.4).
- Seeded `SponsorshipType` lookup values.
- Sample `SponsorshipRequest` rows covering **every** status (Draft, PendingManagerApproval,
  PendingFinanceReview, Approved, Rejected, Cancelled) with matching `WorkflowHistory`.

**Out:** UI (T3.x); the request/workflow use-cases themselves (T2.x) — seeding writes data directly
through the model/EF, not through the (not-yet-built) endpoints.

## References
- HLD §11 (seed data). Brief: test login accounts deliverable. Clarifications: C2 (types), D1 (history).
- `docs/best-practices/security-best-practice-rules.md` (seed creds are test-only; documented, not secret-bearing).

## Roadmap (high-level)
1. Implement an idempotent seeder (checks existence before inserting) invoked after migration.
2. Seed roles (if not already) + 4 users with hashed passwords via Identity.
3. Seed sponsorship types.
4. Seed sample requests across all six statuses, each with a coherent `WorkflowHistory` trail.
5. Document the credentials placeholder for the README (final wording in T4.4).

## Acceptance Criteria (Definition of Done — also workflow.md §6)
- [ ] Build/format/test clean.
- [ ] Fresh `docker compose up` yields a DB with 4 role accounts, types, and requests in every status.
- [ ] Seeder is idempotent (running twice does not duplicate).
- [ ] Test credentials documented (or staged for T4.4); no production secrets.

## Test Expectations
- Integration (Testcontainers): after seeding, assert one user per role exists and at least one
  request exists in each status with consistent history.

## Best-Practices Checklist (subset)
- [ ] Idempotent, deterministic seeding.
- [ ] Passwords hashed via Identity; credentials are clearly test-only.
