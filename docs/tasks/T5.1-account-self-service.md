# T5.1 — Account self-service (profile + password)

**Branch:** `feat/T5.1-account-self-service`  ·  **Depends on:** T1.2, T3.1  ·  **Status:** ⬜

## Context
Authenticated users can update their own account: display name + department, and their password.
Profile management was deferred to the backlog after T3.1 shipped a stub page; this task delivers it.
The hard part is password change: access tokens are stateless short-lived JWTs, so a password change
must (a) immediately invalidate every previously-issued access token and (b) revoke all refresh
tokens, while keeping the current device signed in. We do this with the ASP.NET Identity
`SecurityStamp`: it is embedded as a JWT claim and validated on every request.

## Scope
**In:**
- `/me` self-service endpoints (authenticated, **any role**, **own account only** — no admin/user-id path):
  - `PUT /me/profile` — update `DisplayName` (required) + `Department` (optional); returns updated
    `UserProfileResponse`. Email is read-only here.
  - `PUT /me/password` — `{ currentPassword, newPassword }`; verifies current password and enforces
    the Identity password policy via `UserManager.ChangePasswordAsync` (rehash + automatic
    `SecurityStamp` rotation).
- **Security-stamp validation:** add a `security_stamp` claim to the JWT at issue time
  (`JwtTokenService` + `IssueTokensAsync`); on JWT bearer `OnTokenValidated`, reject (401) when the
  claim ≠ the user's current `SecurityStamp`. Kills all pre-change access tokens immediately.
- **Password-change session handling:** after a successful change → revoke **all** of the user's
  refresh tokens → issue a fresh access token + new refresh cookie for the **current** device
  (caller stays logged in; all other sessions die on next request/refresh). Endpoint returns a
  `LoginResponse` and sets the refresh cookie, mirroring `/auth/login` and `/auth/refresh`.
- Frontend: replace `ProfileStubPage` with a real Profile page — a profile-details form
  (DisplayName, Department; Email read-only) and a separate change-password form (current / new /
  confirm), Zod-validated, following `LoginPage` patterns. New `account-api.ts`
  (`updateProfile`, `changePassword`); on password success, store the returned access token and
  refresh auth context so the header reflects the new name without a re-login.

**Out:** admin-managed user CRUD or role changes (a user cannot change their own role/email — B6,
one role per user); email change / verification; password-reset-by-email / forgot-password flow;
MFA; profile avatar/photo; self-registration (excluded — B6).

## Roadmap (high-level steps, not full code)
1. Add `security_stamp` to access-token claims in `JwtTokenService`; pass `user.SecurityStamp` from
   `IssueTokensAsync`. Validate it in `ConfigureJwtBearerOptions` `OnTokenValidated` (lookup user,
   compare stamp, `context.Fail` on mismatch).
2. Extend `IAuthService` with `UpdateProfileAsync` and `ChangePasswordAsync`; implement in
   `AuthService` (ChangePassword → revoke all refresh tokens for the user → re-issue tokens for the
   current device).
3. Add request models + validators (`UpdateProfileRequest`, `ChangePasswordRequest`); map endpoints
   under `/me` in `AuthEndpoints` (or a new `AccountEndpoints`), `RequireAuthorization()`.
4. Frontend: `account-api.ts`, replace `ProfileStubPage` with profile + password forms, Zod schemas;
   wire success handling (store new token, re-fetch `/me`).
5. Tests (below). Update `.env.example` only if new config is introduced (none expected).

## Acceptance Criteria (Definition of Done — also workflow.md §6)
- [ ] Build/format/test clean (warnings = errors); ESLint/Prettier clean.
- [ ] `PUT /me/profile` updates DisplayName + Department; reflected in `GET /me`. Email unchanged.
- [ ] `PUT /me/password` with correct current password succeeds; wrong current password → 400; weak
      new password → 400 with policy details.
- [ ] After password change: previously-issued access tokens → 401; all prior refresh tokens
      revoked; the current device receives working new tokens and stays signed in.
- [ ] All endpoints operate on the caller's own account only (no user-id in path); unauthenticated → 401.
- [ ] No secrets in the repo.

## Test Expectations
- Unit: profile/password validators; JWT issuance includes `security_stamp`.
- Integration (Testcontainers): profile update reflected in `/me`; password change → old access
  token rejected (401), old refresh rejected, new tokens work, current session stays valid; wrong
  current password → 400; a user cannot affect another user's account.
- Frontend: validation + happy path for both forms; header reflects updated name after save.
