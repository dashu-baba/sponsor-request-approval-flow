# T5.2 — Admin user management (list + create)

**Branch:** `feat/T5.2-admin-user-management`  ·  **Depends on:** T1.2, T3.4  ·  **Status:** ⬜

## Context
Users are currently created **only** by the T1.3 seeder (`UserManager.CreateAsync` +
`AddToRoleAsync`). There is no in-app way to onboard a new user, so a SystemAdmin cannot add a
colleague without a redeploy/reseed. Clarification B6 keeps **self-registration disabled** and
**one role per user**, which makes user creation an explicitly admin-only operation — exactly the
gap this task fills. T5.1 deliberately scoped *admin-managed user CRUD* out of self-service; this is
its complement.

Scope is intentionally narrow: **see the user list** and **create a user**. Editing, role changes,
password resets, and deactivation are out (see Out + backlog) so the first slice stays small and
reviewable.

## Scope
**In:**
- **Backend — admin user endpoints** under `/users`, all `RequireAuthorization(AuthorizationPolicies.SystemAdmin)`
  (mirrors `SponsorshipTypeEndpoints` admin pattern):
  - `GET /users` — list all users as `UserSummaryResponse` (`Id`, `Email`, `DisplayName`,
    `Department`, `Role`), ordered by `Email`. User population is small (seed = 4 + onboarded), so
    return the full list; **no server-side paging yet** (align with T4.1 list conventions if/when
    the set grows — noted, not built here).
  - `POST /users` — create a user: `{ email, displayName, department?, role, initialPassword }`.
    Validates, creates via `UserManager.CreateAsync(user, initialPassword)` (Identity password
    policy enforced) with `EmailConfirmed = true`, then `AddToRoleAsync(user, role)`. Returns
    `201 Created` + `UserSummaryResponse` with `Location: /users/{id}`.
- **Validation** (`CreateUserRequest` validator): `email` required + well-formed + unique
  (duplicate → **409 Conflict**, ProblemDetails); `displayName` required (≤ existing display-name
  limit, match seed/`ApplicationUser` constraints); `role` required and **∈ `Roles.All`** (exactly
  one; reject unknown/empty → 400); `department` optional; `initialPassword` runs through the
  Identity password policy (weak → 400 with policy detail, same shape as T5.1 password errors).
- **Service layer**: extend `IAuthService` (or a focused `IUserAdminService`) with
  `ListUsersAsync()` and `CreateUserAsync(...)`; wrap create + role-assign so a failed role
  assignment does not leave a roleless user (delete-on-failure or transactional cleanup).
- **Frontend — Users admin page** in the SystemAdmin area (lives beside sponsorship-type admin from
  T3.4):
  - Route `/admin/users` (add to `role-nav.ts` SystemAdmin nav, next to Sponsorship Types).
  - A users **table** (Email, Display Name, Department, Role) using the existing list/query +
    `QueryStatePanel` loading/empty/error patterns.
  - A **Create user** modal (mirror `RequestFormModal` / sponsorship-type create): email, display
    name, department, **role `<select>` bound to `Roles.All`**, initial password; Zod-validated;
    on success invalidate the users query and toast. Surface 409 (duplicate email) and 400
    (password policy) inline.
  - New `users-api.ts` (`listUsers`, `createUser`) **or** extend `admin-api.ts`, calling through the
    shared `apiJson`/`apiFetch` client (paths stay unprefixed; client adds `/api` per T4.5).

**Out:**
- Edit user (display name / department / email) → backlog.
- **Role change** after creation (B6 = one role/user; changing roles needs its own rules) → backlog.
- Deactivate / delete / lock user → backlog.
- Admin-initiated password **reset**, and **force-change-on-first-login** (no “must change password”
  flag or login interception exists yet; revisit with T5.1’s SecurityStamp work) → backlog.
- Email invitation / verification flow, self-registration (excluded — B6).
- Bulk import; per-user audit log of admin actions.

## Design decisions (call out for review)
- **Initial password set by the admin** (MVP) rather than a system-generated temp password the user
  must rotate. *Why:* there is no “force password change” flag or first-login interception in the
  codebase yet, and no email channel to deliver a temp secret. Admin-set initial password is the
  smallest secure slice; **force-change-on-first-login is deferred to backlog** and naturally pairs
  with T5.1’s SecurityStamp work. Flagged so the human can decide to pull it in now.
- **No paging on `GET /users`** — the user set is tiny; adding paging now is premature. Align with
  T4.1 list pagination if the population grows.
- **One role, validated against `Roles.All`** — keeps B6 invariant in code, not just convention.

## Roadmap (high-level steps, not full code)
1. Add `UserSummaryResponse` + `CreateUserRequest` models and a FluentValidation validator
   (email/displayName/role-in-`Roles.All`/password-policy).
2. Add `ListUsersAsync` / `CreateUserAsync` to the service layer; implement with `UserManager`
   (create → `AddToRoleAsync`; clean up on role-assign failure). Map Identity errors → ProblemDetails
   (duplicate email → 409, weak password → 400).
3. Add `UserEndpoints` (`/users` group): `GET /` + `POST /`, both
   `RequireAuthorization(AuthorizationPolicies.SystemAdmin)`; register in the endpoint map.
4. Frontend: `users-api.ts` (or extend `admin-api.ts`) + Zod schemas; `/admin/users` route + nav
   entry; users table + Create-user modal with role select bound to `Roles.All`.
5. Tests (below). No new config expected (`.env.example` unchanged).

## Acceptance Criteria (Definition of Done — also workflow.md §6)
- [ ] Build/format/test clean (warnings = errors); ESLint/Prettier/typecheck clean.
- [ ] `GET /users` as SystemAdmin returns all users with role; as any non-SystemAdmin → **403**;
      unauthenticated → **401**.
- [ ] `POST /users` as SystemAdmin creates a user with exactly one valid role; the new user can log
      in with the initial password and `GET /me` reflects the assigned role/profile.
- [ ] Duplicate email → **409**; unknown/empty role → **400**; weak initial password → **400** with
      policy detail; missing required fields → **400**. All errors are ProblemDetails.
- [ ] Non-SystemAdmin cannot reach create (403); B6 upheld — exactly one role, no self-registration
      path introduced.
- [ ] Frontend: SystemAdmin sees `/admin/users` (table + create modal); create success refreshes the
      list; 409/400 shown inline. Non-admins do not see the nav entry/route.
- [ ] No secrets in the repo.

## Test Expectations
- **Unit:** `CreateUserRequest` validator (email format/required, role ∈ `Roles.All`, password
  policy delegation, optional department).
- **Integration (Testcontainers):** SystemAdmin lists users; SystemAdmin creates user → appears in
  list → new user logs in (`/auth/login`) and `/me` shows the role; duplicate email → 409; invalid
  role → 400; non-SystemAdmin (Requestor/Manager/Finance) → 403 on both endpoints; unauthenticated
  → 401.
- **Frontend:** users table renders rows + empty/error states; create-modal validation (required,
  password); happy path invalidates and refetches; 409 duplicate-email surfaced on the email field.
