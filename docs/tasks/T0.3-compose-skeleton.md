# T0.3 — Docker-Compose walking skeleton

**Branch:** `chore/T0.3-compose-skeleton`  ·  **Depends on:** T0.2  ·  **Status:** ⬜

## Context
Make the whole system runnable end-to-end with one command before any feature exists: API (with a
health endpoint) + Postgres + MinIO + nginx (serving the SPA + proxying `/api`) + a one-shot
`migrator` service placeholder. This de-risks deployment and gives every later task a working base.

## Scope
**In:**
- Multi-stage `Dockerfile` for the API (non-root runtime) and for the SPA build (served by nginx).
- `Api` exposes `GET /health` (liveness) using ASP.NET health checks.
- `docker-compose.yml` with services: `db` (postgres:17, healthcheck `pg_isready`, named volume),
  `minio` (named volume), `migrator` (one-shot; placeholder until migrations exist in T1.1),
  `api` (`depends_on` db healthy + migrator completed), `nginx` (publishes 80, serves SPA, proxies
  `/api` and the docs path).
- Internal network: only `nginx` publishes a port; db/minio/api are internal.
- `.env.example` with all config keys (DB, MinIO, JWT placeholder) — no real secrets.
- `docs/deploy.md` skeleton: prerequisites, env setup, `docker compose up`, smoke check.

**Out:** TLS/Let's Encrypt + production hardening (T4.3), real migrations (T1.1 wires the migrator),
real SPA pages (Epic 3), auth (T1.2).

## References
- HLD §3 (system context), §10 (deployment topology + migrator pattern).
- `docs/best-practices/deployment-docker-best-practice-rules.md` (multi-stage, non-root, healthchecks,
  migrator one-shot, internal network, named volumes).

## Roadmap (high-level)
1. Add `GET /health` to the API via `AddHealthChecks()` + `MapHealthChecks("/health")`.
2. Write the API multi-stage Dockerfile (build → publish → slim runtime, non-root user).
3. Write the SPA Dockerfile (build with a pinned **Node 24** image, e.g. `node:24.x.x-slim` — never
   `node:22`/`latest`, per workflow.md §5 — → serve `dist/` via nginx) + nginx config (SPA + `/api`
   + docs proxy).
4. Author `docker-compose.yml` with the 5 services, healthchecks, `depends_on` conditions, named
   volumes, internal network, single published port on nginx.
5. Add `migrator` as a one-shot service that will run `dotnet ef database update` (no-op/placeholder
   until T1.1 adds the DbContext + migration).
6. Add `.env.example`; ensure compose reads from it.
7. `docker compose up` brings the stack healthy; `curl` the health endpoint through nginx succeeds.
8. Write `docs/deploy.md` skeleton.

## Acceptance Criteria (Definition of Done — also workflow.md §6)
- [ ] `docker compose up` starts all services; `db` and `api` report healthy.
- [ ] Health endpoint reachable through nginx (e.g. `http://localhost/api/health` or `/health`).
- [ ] SPA shell served by nginx at `/`.
- [ ] API image runs as non-root; only nginx publishes a port.
- [ ] `.env.example` present with no real secrets; compose uses it.
- [ ] `docs/deploy.md` skeleton committed.

## Test Expectations
- Manual: `docker compose up` + health curl succeeds (capture in PR evidence). No automated tests
  required for infra in this task.

- [ ] Healthchecks + correct `depends_on` conditions (§9, §11); named volumes (§12).
- [ ] No secrets baked into images/compose; `.env.example` only (§6–7).
