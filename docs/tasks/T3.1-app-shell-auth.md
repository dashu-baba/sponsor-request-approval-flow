# T3.1 — Frontend app shell + auth flow

**Branch:** `feat/T3.1-app-shell-auth`  ·  **Depends on:** T1.2  ·  **Status:** ⬜

## Context
The React foundation: routing, providers, the typed API client, and the auth flow (login, token/
refresh handling, role context, route guards). Every later UI feature plugs into this shell.

## Scope
**In:**
- App shell: router, layout, `QueryClient` provider, auth/role context.
- Login page; auth flow against `/auth/login` + `/auth/refresh` + `/me`; access token in memory,
  refresh via httpOnly cookie (no tokens in localStorage).
- Role-aware route guards; a typed API client + base query hooks; Zod schemas for auth responses.
- Loading/empty/error/success handling baseline; error boundary around major sections.

**Out:** request/approval/admin feature views (T3.2–T3.4); backend (Epic 1/2).

## References
- HLD §9 (frontend structure), §2 (frontend stack). Clarifications B6 (no self-registration → login only).
- `docs/best-practices/react-best-practice-rules.md` (§10 server vs UI state, §11 validate external data, §16 four states, §18 context, §22 strict TS, §23 error boundaries).
- `docs/best-practices/security-best-practice-rules.md` (§4 no tokens in localStorage).

## Roadmap (high-level)
1. Set up router + providers + layout; auth/role context.
2. Build the typed API client (handles 401 → refresh → retry; logout on refresh failure).
3. Implement login page + `/me` bootstrap; store access token in memory.
4. Add role-aware route guards; Zod-validate auth responses.
5. Establish loading/empty/error/success + error-boundary patterns.

## Acceptance Criteria (Definition of Done — also workflow.md §6)
- [ ] Typecheck/ESLint/Prettier/build clean.
- [ ] Login works against the API; refresh-on-401 works; logout clears session.
- [ ] Guards redirect unauthenticated users; role context available to features.
- [ ] No tokens in localStorage.

## Test Expectations
- Component tests (RTL): login form behavior; guard redirects; the four UI states on a sample query.

## Best-Practices Checklist (subset)
- [ ] Strict TS, no `any`; server state via TanStack Query (react §22, §10).
- [ ] Validate external data with Zod (react §11); tokens handled securely (security §4).
