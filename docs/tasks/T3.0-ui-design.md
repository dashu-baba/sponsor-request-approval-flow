# T3.0 — UI/UX design system + static prototype

**Branch:** `feat/T3.0-ui-design`  ·  **Depends on:** T0.2  ·  **Status:** ✅

## Context
Establish the agreed look-and-feel before wiring real data, so the frontend feature tasks
(T3.1–T3.4) implement a signed-off design instead of improvising. Visual direction is captured in
`docs/ui-design.md` and the clickable HTML prototype under `docs/ui-mockups/`.

## Scope
**In:**
- Finalize **`docs/ui-design.md`**: screen inventory, information architecture / navigation,
  per-screen layouts, component mapping to **shadcn/ui** (Tailwind + Radix), status-badge system for
  the six request statuses, loading/empty/error/success (toast) patterns, responsive + accessibility
  notes, design tokens (color, typography, spacing), and the **draft-visibility rule** (draft-related
  UI appears **only for Requestor** — no other role sees drafts in lists, metrics, or filters; B5).
- **Static HTML prototype** in `docs/ui-mockups/` (mock data, no backend): `index.html` entry,
  login, four role dashboards, unified request detail (role query param), sponsorship-type admin,
  loading/error/409 state screens — enough to click through and sign off the visual design.
- Map mockup tokens from `app-shell.css` to the shadcn theme that T3.1+ will implement in React.

**Out:** real API wiring + auth (T3.1), feature behavior/business rules (T3.2–T3.4), backend.

## Roadmap (high-level)
1. Finalize `docs/ui-design.md` from mockup outcomes (screens, tokens, components, states, IA).
2. Ensure `docs/ui-mockups/` covers all key flows; keep `index.html` as the screen map.
3. Document shadcn/ui component mapping and theme tokens for implementers (T3.1–T3.4).
4. Visual sign-off; note any follow-ups.
