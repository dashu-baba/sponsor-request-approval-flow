# T3.0 — UI/UX design system + static prototype

**Branch:** `feat/T3.0-ui-design`  ·  **Depends on:** T0.2  ·  **Status:** ⬜

## Context
Establish the agreed look-and-feel before wiring real data, so the frontend feature tasks
(T3.1–T3.4) implement a signed-off design instead of improvising. The design direction is explored
collaboratively via the Visual Companion and captured in `docs/ui-design.md`; this task turns that
into a real, reviewable static prototype.

## Scope
**In:**
- Finalize **`docs/ui-design.md`**: screen inventory, information architecture / navigation,
  per-screen layouts, component mapping to **shadcn/ui** (Tailwind + Radix), status-badge system for
  the six request statuses, the loading/empty/error/success patterns, responsive + accessibility
  notes, and design tokens (color, typography, spacing).
- A **static prototype** in the frontend (mock data, no backend): login, requestor dashboard +
  request form, manager/finance approval queue + review detail, system-admin console — enough to
  click through and sign off the visual design.

**Out:** real API wiring + auth (T3.1), feature behavior/business rules (T3.2–T3.4), backend.

## References
- HLD §9 (frontend structure), §2 (stack). `docs/ui-design.md` (produced from the Visual Companion session).
- `docs/best-practices/react-best-practice-rules.md` (§1 focused components, §16 four states, §19 accessibility, §21 consistent styling, §22 strict TS).

## Roadmap (high-level)
1. Finalize `docs/ui-design.md` from the Visual Companion outcomes (screens, tokens, components, states).
2. Add shadcn/ui + Tailwind to the frontend scaffold; set up the theme/tokens.
3. Build the static screens with mock data and shared components (badges, tables, forms, dialogs).
4. Wire simple client-side navigation so the screens are clickable for review.
5. Visual sign-off; note any follow-ups.

## Acceptance Criteria (Definition of Done — also workflow.md §6)
- [ ] Typecheck/ESLint/Prettier/build clean.
- [ ] `docs/ui-design.md` complete and matches the prototype.
- [ ] Static prototype renders all key screens with mock data and is navigable.
- [ ] shadcn/ui theme + status-badge system + the four UI states are established as reusable components.

## Test Expectations
- Component render smoke tests for the shared components; the primary gate is visual review.

## Best-Practices Checklist (subset)
- [ ] Focused components; consistent single styling approach; accessible semantics (react §1,21,19).
- [ ] Strict TS; the four UI states modeled as reusable patterns (react §22,16).
