# T2.3 — Sponsorship-type admin CRUD

**Branch:** `feat/T2.3-sponsorship-types`  ·  **Depends on:** T2.1  ·  **Status:** ⬜

## Context
System Admin manages the sponsorship-type lookup used by requests. A small, self-contained admin
capability. Can be built in parallel with T2.2/T2.4 once T2.1 lands.

## Scope
**In:**
- CQRS handlers + DTOs (records) + validators for sponsorship types.
- Endpoints `GET/POST/PUT/DELETE /sponsorship-types`, restricted to `SystemAdmin` (B3).
- Sensible delete semantics: `DELETE` soft-disables a type by setting `IsActive = false`, preserving
  any existing requests that reference it while preventing future request creation with that type.

**Out:** request features (T2.1), workflow (T2.2), frontend admin console (T3.4).

## References
- HLD §8 (API), §5 (SponsorshipType). Clarifications C2 (required managed lookup), B3 (SystemAdmin scope).
- `docs/best-practices/aspnetcore-best-practice-rules.md` (§1, §4–6, §11–12).

## Roadmap (high-level)
1. DTOs + validators (unique name, required fields).
2. Implement list/create/update/delete handlers; enforce referential safety on delete.
3. Add endpoints behind a `SystemAdmin` policy; problem-details errors.

## Acceptance Criteria (Definition of Done — also workflow.md §6)
- [ ] Build/format/test clean.
- [ ] CRUD works and is restricted to SystemAdmin (others → 403).
- [ ] Deleting a type that is referenced by requests is handled safely by soft-disabling it, documented.

## Test Expectations
- Unit: validators (unique/required), delete-in-use rule.
- Integration: admin CRUD happy path; non-admin forbidden; delete-in-use behavior.

## Best-Practices Checklist (subset)
- [ ] Role policy enforced server-side (security §1–2).
- [ ] Thin endpoints; record DTOs; correct status codes (aspnetcore §1,5,6).
